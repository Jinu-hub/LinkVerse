## 開発とサービス運営で使っている主なツール

開発を続けていくと、技術スタックと同じくらい大事なものがある。  
それは、どのツールやサービスを組み合わせて、回しているかという点だ。

フレームワークや言語、データベースといった技術スタックとは別に、  
サービスを作って運営して際には、それらを巡るツールもかなりの比重を占める。

AIにコードを聞き、  
デプロイ環境を管理し、  
メールを送り、  
外部APIをつなぎ、  
自動化フローを組み、  
データを整理し、  
アイデアを記録し、  
決済や認証まで載せる。

これらが重なって、ひとつのサービス開発の流れになる。

今回の記事では、2026年現時点で、開発とサービス運営に使っている主なツールとサービスを整理してみたい。  
技術スタックそのものをまとめるというより、  
個人開発者が実際にサービスを作り運営するために、どんなツールを組み合わせているかの記録に近い。

もちろんこのリストは固定ではない。  
来年や再来年には、今使っているものの一部は自然に外れていくかもしれないし、まだ軽く触っているだけのツールが、より中心の位置に来るかもしれない。

その意味で、この記事は「正解リスト」というより、2026年現時点の、自分がどう開発し、サービスを運営しているかを残したスナップショットだ。

---

## 1. AI / 開発補助ツール

### Cursor AI

いまの開発で最もよく使うツールのひとつだ。  
単なるコード補完を超えて、プロジェクト全体の文脈を踏まえて修正の方向を決めたり、リファクタリングを進めたりするのに役立つ。

特に一人で開発するときは、「横で一緒にコードを見てくれる人」がいる感覚が大きい。  
最終判断は人がする必要があるが、繰り返しの実装や構造の整理では、生産性をかなり上げてくれる。

### ChatGPT

アイデア整理、設計の確認、コードの方向性、文章づくり、プロンプト設計など、ほぼ全方位で使っている。

開発ツールでもあるが、同時に思考を整理するツールに近い。  
機能を作る前に概念をまとめたり、データの流れを固めたり、ブログの下書きを作ったりするときにもよく使う。

自分にとって ChatGPT は、単なる Q&A というより、プロジェクトの方向と構造を一緒に点検する補助パートナーに近い。

### Gemini

ChatGPT とは違う視点の回答を比べたり、特定の作業について別の見方を得るために使う。

AI ツールはそれぞれ回答のスタイルと強みが違うので、ひとつに固定するより、状況に応じて比べる方がよいと感じている。

### Claude Code

コード作業に特化した流れを組むときに注目しているツールだ。  
複雑なコードベースを扱うときや、複数ファイルをまとめて直す作業では、活用の余地が大きいと見ている。

まだすべてを任せる段階ではないが、AI ベースの開発ワークフローでは重要な軸になりうると思っている。

---

## 2. AI API / モデル / 埋め込み

### OpenAI API / Models

サービス内部で AI 機能を実装するために使う中核の API だ。  
チャットボット的な使い方だけでなく、レポート生成、要約、分類、タグ付け、構造化データの生成など、さまざまな作業に使っている。

自分が作っているサービスは、コンテンツとデータの解釈が大きな比重を占めるので、OpenAI API は小さな付加機能ではなく、サービス構造の重要な一部になっている。

### OpenAI Embeddings API

テキスト間の類似度やベクトル検索のために使っている。  
たとえば過去のレポートと現在のレポートがどれだけ似ているかを計算したり、特定のテーマに関連するコンテンツを見つけたりできる。

コンテンツが積み上がるほど、単純な検索より「似た文脈を見つける機能」の価値は大きくなると思う。  
その意味で Embeddings API は、長期的にはかなり重要な役割を担う可能性がある。

---

## 3. デプロイ / インフラ / 運用

### Vercel

フロントエンドサービスのデプロイに使っている。  
素早くデプロイして結果を確認できる点が大きな利点だ。

個人開発者や小さな MVP では、デプロイが複雑になると開発のリズムが切れやすい。  
その点で Vercel は、速い実験とリリースに合うツールだと感じている。

### Cloudflare

ドメイン、DNS、Workers、Pages など、いくつかの領域で使っている。  
CDN や DNS だけではなく、だんだん広い範囲のアプリケーションインフラを担えるプラットフォームとして見ている。

個人開発者がコストと複雑さを抑えながらサービスを運営するうえで、よい選択肢のひとつだと思う。

### Cloudflare Workers

サーバーレスで軽いバックエンドロジックを動かすときに使う。  
API のルーティング、プロキシ、簡単なサーバー処理などに向いている。

重いサーバーを自分で運用しなくても、比較的早く機能を足せる点が利点だ。

### Cloudflare Pages

静的サイトやフロントエンドプロジェクトのデプロイに使える。  
Cloudflare の中で Workers と組み合わせれば、比較的シンプルな構成でもかなりのことが作れる。

### Wrangler

Cloudflare Workers の開発とデプロイに使う CLI だ。  
ローカル開発、型生成、デプロイなど、Cloudflare ベースのプロジェクトでは欠かせなくなっている。

最初は設定が少し面倒に感じることもあるが、慣れると Workers ベースの開発の流れはかなりすっきり進められる。

### Namecheap

ドメインの取得と管理に使っている。  
サービスを作っていくと、ドメインは単なる住所以上の意味を持つ。

個人プロジェクトやブランドを育てるときは、どんな名前にするか、どの拡張子にするかも、それなりに大事な判断になる。

---

## 4. データベース / バックエンド基盤

### Supabase

データベース、認証、ストレージ、ベクトル検索などを提供するバックエンドプラットフォームとして使っている。

個人開発者にとって、バックエンドの多くを自分で組まなくてよい点は大きい。  
PostgreSQL ベースであることも好みだ。

ベクトル検索までまとめて使えるので、AI ベースのサービスやコンテンツ検索とも相性がよい。

---

## 5. 自動化 / ワークフロー

### n8n

複数の API やサービスをつなぎ、自動化フローを組むのに使っている。

特に AI Agent のパイプラインを組むときに有用だ。  
データを取り、LLM に渡し、結果をパースして、再び DB に保存する流れを、視覚的に確認できる点が大きい。

すべてをコードだけで書くこともできるが、n8n は中間のデータと流れを目で追える利点がある。  
個人的には、AI ベースの自動化作業を設計するときに、かなり重要なツールだと感じている。

### Google Drive

ファイルの保存、共有、資料管理に使う。  
開発そのものというより、運用とドキュメントの面でよく使うようになる。

### Google Sheets

軽いデータ整理、テストデータの管理、運用用テーブルなどに使える。  
本格的なデータベースに手を入れる前に、一時的に構造を固めたり、人が確認すべきデータを整理したりするときに便利だ。

---

## 6. 認証 / OAuth / Google 連携

### Google Cloud Console

Google API を使うために必要な設定を管理する場所だ。  
OAuth クライアントの設定、API の有効化、認証情報の管理などをここで行う。

最初は少し面倒に感じることもあるが、Gmail、Drive、Calendar などの Google サービスと連携するなら、避けて通れない領域だ。

### Google OAuth

外部サービスと Google アカウントをつなぐ認証方式として使っている。  
特に n8n や自前のサービスから Gmail、Drive などの API をつなぐときに必要になる。

認証と権限は、サービス運営では思った以上に重要な領域だ。  
機能そのものより、ユーザーが安全につなげて使える構造を作ることが大事だ。

---

## 7. メール / 決済 / ビジネス機能

### Resend

サービスからメールを送るために使う。  
会員登録、通知、レポート配信、トランザクションメールなど、さまざまな場面で使える。

メール機能は見た目は単純だが、実際の運営ではとても重要な要素だ。  
安定して送れ、開発者が扱いやすい API である点で、Resend はよい選択肢だと感じている。

### Stripe

決済機能を載せるために検討・利用しているサービスだ。  
個人開発者や小さなサービスがグローバルな決済をつけるとき、現実的な選択肢のひとつだ。

最初から複雑な課金体系を作る必要はないが、いつか有料化するなら、決済インフラをどうつなぐかは早めに考えておく価値がある。

---

## 8. コミュニケーション / 協業

### GitHub

コードの保管場所であり、開発作業の中心でもある。  
バージョン管理だけでなく、イシュー、PR、デプロイ連携など、開発の流れの多くが GitHub を中心に回る。

個人プロジェクトでも、GitHub は単なるリポジトリではなく、作業の履歴と判断の記録になる。

### Slack

業務コミュニケーションと通知の流れで使っている。  
Slack のデータは、AI ベースのレポートや業務の要約自動化でも活用の余地が大きい。

チームの動きを整理したり、特定期間にどんな論点があったかを要約したりするとき、Slack のデータは重要な入力になりうる。

### Telegram

軽い通知、個人用ボット、自動化メッセージの受信などに使える。  
Slack より個人的な通知チャネルとして使いやすい点が利点だ。

---

## 9. ドキュメント / ナレッジ管理

### Obsidian

個人のナレッジ管理と思考の整理に使っている。  
つながった考えを記録し、アイデアの流れを保つのに向いている。

開発プロジェクトでは、単なる文書より、思考の変化とつながりを残すことが大事になるときがある。  
その点で Obsidian は、単純なメモアプリより個人のナレッジベースに近い。

### Notion

企画文書、設計文書、プロジェクトの整理、運用ドキュメントなどを管理するのに使う。  
共有したり、整った形で見るには Notion の方が扱いやすい。

Obsidian が個人的な思考の流れを載せるツールなら、Notion は、もう少し整えた文書と運用資料を管理するツールに近い。

---

## 10. 外部データ / 金融データ

### Twelve Data

市場データや金融データを取得するために使っている。  
株価、指数、為替など、外部データをサービスにつなぐときに使える。

無料プランや料金体系によって、使えるシンボルや機能に制限があるので、実際のサービスに載せるときは、コストと制限を一緒に考える必要がある。

### Fear & Greed API

市場心理を示す指標として使える。  
価格データだけでは市場の雰囲気を説明しにくいときがあり、こうした心理指標はダッシュボードやレポートの補助データとしてよい。

### Yahoo Finance / Finance Data Source

市場データの参照や補助的な確認に使っている。  
本番で安定して使うには別途検討が必要だが、市場の流れを把握するデータソースとしては、いまも有用だ。

---

## 11. 分析 / モニタリング / デザイン（候補）

まだメインツールとして完全に定着したわけではないが、これからもっと使ってみたいツールもある。

### Figma

UI 設計や画面構成を固めるときに使えるデザインツールだ。  
個人開発者がすべてのデザインを細かく作る必要はないが、主要画面の構造やユーザーの流れを決めるのには役立つ。

サービスが大きくなるほど、いきなりコードに入る前に画面の方向を整理する工程は、より重要になるかもしれない。

### PostHog

ユーザー行動やプロダクト分析に使える。  
訪問者数を見るだけでなく、どの機能が使われ、どこで離脱するかを確認できる。

MVP の段階では分析を載せすぎると負担になることもあるが、実際のユーザーがついてきたら、重要なツールになりうる。

### Google Analytics

サイトの訪問者数、流入経路、ページビューなどを確認する基本的な分析ツールだ。  
ブログやランディングページ、公開サービスの運営では、いまも有用だ。

### Canva

簡単な画像、サムネイル、資料、SNS 用画像などを作るときに使える。  
本格的なデザインツールまでは不要でも、素早くビジュアルを作りたいときの選択肢としてよい。

---

## まとめ

いま使っているツールを並べてみると、「開発ツールをたくさん使っている」という話だけではない。

むしろ、ひとつのサービスを作るために必要な流れが、少しずつ見えてくる。

AI で思考と実装を補い、  
Supabase と Cloudflare でバックエンドとインフラを組み、  
Vercel で素早くデプロイし、  
n8n で自動化のパイプラインを作り、  
OpenAI API と Embeddings で知的な機能を載せ、  
Google、Slack、GitHub、Notion、Obsidian などで運営と記録をつなぐ。

個人開発者にとって大事なのは、すべてを自分で作ることではない。  
必要なツールをうまく組み合わせて、一人でもサービスを作り運営できる構造をつくることだ。

自分にとってこれらのツールは、単なる便利機能というより、小さなチームや個人が、もっと広い範囲の仕事をこなすためのレバレッジに近い。

このリストはこれからも変わっていく。  
深く使うものもあれば、自然に外れていくものもあるだろう。

ただ、いまの時点では、このリストが、自分がどう開発し、サービスをどんな方向に作っていくかを示すひとつの記録になれるのではないかと思う。
