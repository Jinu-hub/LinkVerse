## NexLetterのレポート生成構造についての再考

NexLetterを作っていて、最近また考え直していることがある。  
それは、**週次レポートを生成するとき、データをどの単位で扱うか**という問題だ。

現状の構造では、週次レポートは1週間分のデータを一度に取り込んで処理する方式に近い。

GitHubのデータは、このやり方でも比較的いける。  
コミット、PR、Issue、変更ファイル数などはある程度構造化されていて、データ量も予測しやすい。

一方、Slackのデータは違う。

Slackは現場やチームのコミュニケーションの仕方によってデータ量が大きく変わる。  
あるチームは一日にたくさんのやりとりがあるし、あるチームは必要な内容だけ短く残す。  
だから同じ「1週間分のデータ」でも、実際に処理しなければならないトークン量は現場ごとにかなり違う。

このあたりで、`429 Rate limit reached for [モデル名]`のような問題が出てくる可能性がある。

もちろん、Rate Limitそのものを上げる方法もある。  
ただしそれはOpenAI側の判断やアカウントの状態に依存する領域だ。

サービス設計の観点では、単にLimitの増加に頼るより、  
入力データをどう分割し、要約し、標準化し、段階的に処理するかの方が重要になってくる気がする。

---

## これまでの方式の限界

従来の方式は比較的シンプルだ。

```txt
1週間分のRaw Data
        ↓
AI処理Pipeline
        ↓
週次レポート生成
```

MVPの段階では、この方式は悪くない。  
構造が単純で、実装も速い。

ただしデータ量が増え始めると、いくつか限界が見えてくる。

一つ目は**トークン量の不安定性**だ。  
GitHubのデータはある程度予測できるが、Slackのデータはそうではない。  
障害対応、長い議論、リリース準備、企画変更などが重なると、特定の週だけデータ量が急に膨らむことがある。

二つ目は**処理時間の増加**だ。  
1週間分を一度に分析してレポートまで作ろうとすると、最終結果が出るまで時間が伸びる。

三つ目は**再利用性が低い**という点だ。  
Raw Dataを一度処理して終わりにすると、その途中でできた分析結果を別の形で使い回しにくい。

たとえば後から日次レポート、月次レポート、期間ごとの類似イシュー分析をしたくても、またRaw Dataから読み直す可能性が高い。

つまり、従来の方式は素早く始めるには向いているが、データ量が増えたり、レポートの活用範囲が広がったりすると、限界がはっきりしてくる可能性がある。

---

## 改善の方向性：まずDaily Core Dataを作る

今考えている代替案は次のとおりだ。

週次レポートを作るにしても、1週間分を一度に処理しない。  
代わりに毎日データを集め、その日のデータをもとに**Daily Core Data**を作って保存する。

そして週次レポートを生成するときは、Raw Data全体を最初からやり直さず、  
すでに整理された5〜7件のDaily Core Dataを土台に最終レポートを作る。

```txt
Daily Raw Data
        ↓
Daily Core Data生成
        ↓
Daily Core Data保存
        ↓
5〜7件のDaily Core Dataを集める
        ↓
Weekly Report生成
```

週末は通常、業務が多くないので、実際には5件程度のCore Dataが週次レポートの主な材料になる可能性が高い。

ここで大事なのは、Daily Core Dataが完成した日次レポートではないということだ。

Daily Core Dataは、最終ユーザーに見せるための文章というより、  
週次や月次レポートを作るための**中間表現データ**に近い。

言い換えれば、NexLetterのレポート生成の流れは、だいたい次のような方向に変わる。

```txt
Raw Data
        ↓
Core Data
        ↓
Report
```

Raw DataとFinal Reportのあいだに再利用可能な中間データができ、  
この中間データが、その後の拡張の核になりうる。

---

## Daily Core Dataに載せられる情報

現在の週次レポートの構成は、おおよそ次のようになっている。

```txt
Summary
KPI INFO
HIGHLIGHTS
TOPIC
PROGRESS AND ROADMAP
MEMBERS ACTIVITY
ENDING
```

このうち`Summary`と`ENDING`は、最終の週次レポートの段階で書く方が自然だ。

Summaryは一週間全体の流れを見て書く必要があるし、  
Endingもその週の空気感と次の流れをまとめる役割に近いからだ。

一方、下のセクションは日次単位でCore Dataを作っておくのに向いている。

```txt
KPI INFO
HIGHLIGHTS
TOPIC
PROGRESS AND ROADMAP
MEMBERS ACTIVITY
```

たとえば一日分のSlack / GitHubデータを分析して、次のような形に整理しておける。

```txt
Daily Core Data
- その日の主要KPI
- 主なハイライト
- 議論されたトピック
- 進捗と今後の作業
- メンバー別の活動要約
- 関連タグ
- イシュー分類
- データの出所情報
- 使ったモデルとプロンプトのバージョン
```

こうしてできたCore Dataは、週次レポートの材料だけに使われない。

後から特定の期間にどんなイシューが繰り返されたか、  
どんな種類の対応がよく起きたか、  
どのメンバーがどの領域で動いたか、  
特定プロジェクトの流れがどう変わったかを分析する用途にも使える。

---

## 期待できるメリット

この方式の一番大きな利点は、処理単位が小さくなることだ。

1週間分を一度に処理する代わりに、日単位に分けて処理すれば、一回のAI呼び出しで扱うトークン量が減る。  
その分、Rate Limitに引っかかる可能性も下がる。

すでにDaily Core Dataができていれば、  
週次レポート生成の段階でRaw Dataを最初から読み直す必要もない。

整理されたコアデータだけで週の流れを合成できるので、  
レポート生成時間も短くなる可能性が高い。

この構造は拡張性の面でも有利だ。

```txt
Daily Core Data → Daily Report
Daily Core Data 5件 → Weekly Report
Daily Core Data 20〜25件 → Monthly Report
Daily Core Data N件 → 期間別イシュー分析
```

最初は週次レポートのためにDaily Core Dataを作るが、  
あとから同じデータで日次や月次へ広げられる。

TagとClassificationのデータも一緒に保存しておけば、  
単純な要約を超えて、期間をまたいだ類似対応のチェックもしやすくなる。

こう見るとDaily Core Dataは単なるキャッシュではない。  
NexLetterが組織の活動データを覚えて、あらためて解釈するための基盤データになりうる。

---

## Tag / Classificationが開く別の可能性

Daily Core Dataには、TagやClassificationの結果も含める予定だ。

ここで特に面白いのは、Tag / Classificationが単なる分類用メタデータで終わらないということだ。

たとえば週次レポートの中で特定のイシューやトピックが出てきたとき、  
そのイシューに似た対応が過去にいつ起きたかも一緒に示せる。

「今週こういうことがあった」で終わるのではなく、  
「類似の対応は2026年第12週のレポートでも確認できる」という短いメモを添えられる、というイメージだ。

```txt
参考メモ
類似のSlackアラート失敗への対応は、2026年第12週のレポートでも確認できる。
当時は外部APIのRate Limitとリトライ処理のやり方が主な論点だった。
```

この短いメモ一つでも、レポートの性格は変わると思う。

単に現在の状態をまとめた文書ではなく、  
過去の対応履歴と今の問題をつなぐ文書になる。

人は過去に似た問題があったことを忘れやすい。  
特にSlackやGitHubに散らばった記録は、時間が経つと探しにくい。

でもNexLetterがCore Dataをもとに類似イシューを見つけて、  
どの週のレポートを参照すればいいかまで教えてくれるなら、  
レポートは単なる要約ではなく、組織の記憶を取り出す道具になりうる。

結局、TagとClassificationは検索を楽にするための付加情報ではない。

現在のイシューと過去の対応をつなぎ、  
繰り返す問題の流れを把握し、  
レポートにより深い文脈を与えるためのインデックスとして機能する。

---

## 注意すべき点

ただし、この構造が無条件に良いというわけではない。

まず考えなければならないのはコストだ。

これまでは週次レポートを作るタイミングで一度大きく処理すればよかった。  
Daily Core Dataを作り始めると、毎日AI処理が走る。  
その分呼び出し回数が増え、AI利用料も増える可能性がある。

もう一つの問題は開発範囲だ。

既存の週次レポートのpipelineにDaily Core Data生成の段階を足し、  
それを保存するテーブルやデータ構造も必要になる。

日次処理が失敗したときの再実行の仕方、抜けの検知、状態管理もセットで考える必要がある。

重複処理も気にかける。

一つのイシューが月曜から水曜まで続けて議論されたなら、  
各Daily Core Dataには同じイシューが繰り返し入ることがある。

そのとき週次レポートの段階では、単にデータをつなげるのではなく、  
同じ流れを一つの週次トピックにまとめる過程が必要になる。

類似イシューを見つける機能も慎重に扱う必要がある。

TagやClassificationが似ていても、常に同じ文脈のイシューとは限らない。  
だから過去のレポートとつなぐときは「同一の問題」と断定するより、  
「類似の対応履歴がある」程度の表現の方が安全だと思う。

## 週次レポートの段階で必要になる処理

Daily Core Dataを土台に週次レポートを作るなら、最終段階では単純な要約ではなく合成が必要になる。

週次レポート生成の段階では、次のような処理が重要になってくる気がする。

```txt
Weekly Synthesis
- 複数日にまたがる同一トピックのマージ
- 繰り返しイシューの整理
- 重要度の再評価
- 完了 / 進行中 / 保留の切り分け
- メンバー別の活動パターンの整理
- 過去の類似イシューの探索
- 参照できる過去レポートとの接続
- 週次Summaryの生成
- 自然なEndingの執筆
```

特にSlackのデータは断片的な会話が多いので、  
日次では重要に見えた内容が、週次では小さなイシューになることもある。

逆に、一日では目立たなかった内容が、  
数日続くことで重要な流れとして現れることもある。

だからWeekly Reportの段階では、単にDaily Core Dataを足し合わせるのではなく、  
一週間全体の流れをあらためて判断する過程が必要になる。

ここに過去レポートとの接続まで加われば、  
週次レポートは現在の活動を整理する文書から、一歩先に進める。

今起きているイシューが初めてなのか、  
過去にも似た対応があったのか、  
そのときはどう処理したのかを、一緒に示せるからだ。

## メリットとトレードオフの整理

まとめると、この構造のメリットは次のとおりだ。

| 項目 | 期待できる効果 |
|---|---|
| トークン量の削減 | 一度に処理するデータ量が減る |
| Rate Limitリスクの低減 | 分あたりのトークン負荷を下げられる |
| 処理速度の改善 | 週次レポート生成時にすでに整理されたデータを使える |
| レポート品質の向上 | Raw Dataを直接要約するより、構造化データベースの生成がしやすい |
| 日次 / 月次への拡張 | 同じCore Dataを別の期間のレポートにも使える |
| 類似イシュー分析 | Tag / Classificationで繰り返す対応を追える |
| 過去レポートとの接続 | 現在のイシューに似た過去の対応をレポート内で参照できる |
| 再処理のしやすさ | 最終レポートだけ作り直すときにRaw Data全体を読み直さなくてよい |

一方、トレードオフは次のとおりだ。

| 項目 | 考えるべき点 |
|---|---|
| AI利用料の増加 | 毎日AI処理が走るので呼び出しコストが増える可能性がある |
| 開発範囲の拡大 | 既存のpipelineをかなり大きく直す必要がある |
| データ設計 | Core Dataのスキーマを慎重に決める必要がある |
| 重複イシュー処理 | 複数日にまたがる同一イシューをマージするロジックが必要 |
| 類似イシューの判断 | 似たタグでも実際の文脈は違うことがある |
| 運用の複雑さ | 失敗時の再実行、抜けの検知、状態管理が必要になる |

## 現時点での判断

今の段階で、この構造をすぐに完成形として作る必要はないと思う。

ただし方向性自体はかなり妥当に見える。

最初は単にRate Limitの問題を避けるための対応に見えたが、  
もう少し考えると、NexLetterの構造をより安定させる方向でもある。

Raw Dataをそのまま最終レポート生成に流す方式は、素早く始めるにはいい。  
でもデータ量が増え、顧客や現場が多様になるほど、限界ははっきりしてくる可能性が高い。

一方、Daily Core Dataを蓄える方式は、初期実装のコストはかかる。  
AI利用料も増える可能性がある。  
でも長期的には、レポート生成の速度、品質、拡張性、分析のしやすさの面で、より良い土台になりうる。

特にTag / Classificationをもとに、現在のイシューと過去のレポートをつなげられるなら、  
NexLetterのレポートは単なる週次の要約を超えられる。

「今週何があったか」を伝えるだけでなく、  
「過去にも似たことがあったか」、  
「そのときはどう対応したか」、  
「その内容はどのレポートで確認できるか」までつなげられるからだ。

結局大事なのは、単に「一度うまくレポートを作ること」だけではない。

NexLetterが続けて伸びていくには、  
組織の中で毎日起きる活動データを、再利用できる形で積み上げていく必要がある。

その意味でDaily Core Dataは単なる中間成果物ではなく、  
NexLetterがチームの流れを記憶し、解釈するための核となるデータになりうる。

まだ最終決定とは言えないが、  
現時点ではこの方向をかなり真剣に検討する価値があると感じている。
