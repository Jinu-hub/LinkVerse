## 背景

2026年4月29日時点で、自分が進めている主要プロジェクトの状態を一度整理しておきたい。

今進行中のプロジェクトは、それぞれ性質が違う。

LinkVerseは長期的に自分が作るサービスの土台に近いプラットフォームで、  
NexLetterはSlackとGitHubのデータをもとにAIレポートを生成する自動化サービスだ。  
MarketReportは経済と市場に関する分析コンテンツを生成し、それをサービスとして見せるためのプロジェクトだ。

各プロジェクトはまだ完成した状態ではない。  
ただ、ある程度方向が見え始めたものもあれば、今の段階ではっきり見えている課題もある。

この記事は、現時点での進捗と判断を残すための記録だ。

---

## LinkVerse

LinkVerseは現在、ブックマーク機能を中心に実装されている。

ブックマークにはタグとメモを一緒に残せるようにし、今の段階では個人的な知識の整理とリンク管理の基本機能を揃えることに集中した。

ただ、LinkVerseを単なるブックマークサービスだけと考えてはいない。  
長期的には、自分が作る複数のサービスをつなぐプラットフォームの役割を担う予定だ。

個別のサービスが別々に存在していても、  
その中心には自分を紹介し、作ったものを整理し、文章と記録を積み上げていく場所が必要だと考えている。  
なので当分は、LinkVerseのアップデートは次の二つの領域を中心に運用していくつもりだ。

- 自分を紹介する `/orbit`
- 考えと作業を記録する `/blog`

今の段階では、LinkVerseに大きな機能を次々足すより、「自分の活動とプロジェクトを載せる基盤の場所」として整えていくことがより重要だと思っている。

特にブログは単なる宣伝チャンネルではなく、何を作っていて、なぜそう判断したのか、どんな過程を経ているのかを残す記録の場として使っていく。

## NexLetter

NexLetterは、GitHubとSlackのデータを取り込み、AI Agent pipelineを通したあと、週次レポートを作成してメールで送るサービスだ。

現在はSlack Appマーケットのレビューが進行中だ。  
Slack Appが通れば開発を続ける予定だ。

機能の大きな方向性はある程度固まっている。  
ただ、開発を本格的に再開する場合、解決しなければならない課題もはっきり見えている。

最大の課題は、非定型データの扱いだ。

現場ごとにSlackチャットのやり方、業務記録の密度、会話の流れ、用語の使い方がみな違うことがある。  
つまり、同じSlackデータでも、実際にはほぼ別の形の入力データとして扱う部分があると思う。

この点では、まだ十分なテストソースが足りない。

特にトークン量が増えたときの処理の仕方は、もう少し慎重に見る必要がある。  
同じモデルを使っていても、分あたりのトークン処理量が増えるとエラーが出ることもあり、  
実際に `429 Rate limit reached for [モデル名]` のような問題が起きることがある。

もちろんlimitを増やす方法もあるが、それは結局OpenAI側の判断が必要な領域だ。  
サービス設計の視点では、単にlimitの増加に頼るより、入力データをどう分割し、要約し、標準化し、段階的に処理するかの方が重要になってくると思う。

それ以外にも、いくつか課題が残っている。

- レポート作成時に動くRailwayワーカーの安定稼働
- 課金ポリシーの設計
- スムーズな billing システムの流れに関するテスト
- さまざまなSlack利用パターンへの検証
- 大量データ処理時のコストと品質のバランス

NexLetterはアイデアそのものより、運用段階の安定性が重要なサービスだ。

AIがレポートをよく書くことも大事だが、実際の現場のデータを安定して受け取り、処理し、コストを抑えながら、ユーザーが納得できる成果物を継続的に送れる必要がある。

今はSlack Appのレビュー結果を待ちながら、次の開発段階で扱うべき問題を整理している段階だ。

## MarketReport

MarketReportは、ミクロ経済とマクロ経済、そして市場全体に関する分析ソースをもとにレポートを発行するサービスだ。

現在はn8nを使った自動化 pipeline を中心に構築している。  
複数のAI Agentがそれぞれ役割を持ち、入力されたソースを分析・整理し、最終的に読めるレポートの形にする構造だ。

現時点では第1段階の pipeline はほぼ完成に近い。

現時点では3本のレポート pipeline がある程度形を持ち、いまは登録したデータをWebページでどう表現するかを考える段階に移っている。

つまり、これまでは「レポートをどう生成するか」が中心だったなら、これからは「生成したレポートをユーザーがどう読み、探索させるか」が重要になってきている。

現在考えている方向性は次のとおりだ。

- レポート本文をどう読みやすく見せるか
- 要約、SNS用の文、本文、HTML表現をどう分けるか
- 複数レポートの関係をどう見せるか
- 類似度の高いレポート同士をどうつなぐか
- 単なる記事一覧ではなく、市場を理解するための探索構造をどう作るか

今後は pipeline をさらに高度化し、レポートの種類も増やしていく予定だ。

特に vector data を使った類似度の計測は、重要な方向になりそうだ。  
各レポートが単独で存在するのではなく、似たテーマや流れを持つレポート同士とつながれば、より高い水準の分析レポートを生成できるだろう。

たとえば類似度の高いレポートを束ねて、特定の産業や市場の流れに関する、より広い視点のレポートを作れる。  
そうなれば、一つのニュースを要約するレベルを超え、複数の情報の関係をもとに市場を読むサービスに近づける。

ただ、今見えている課題もはっきりしている。

レポートのテキスト自体は、かなり高い水準の内容と可読性を示している。  
しかしそれを補うグラフや図を入れるための素材と、システムとしての構造がまだ十分に整っていない。

市場レポートは文章だけでも伝えられるが、読者がより早く理解するには視覚的な補助が必要だ。

たとえば次のような要素だ。

- 主要指標の変化グラフ
- 企業または産業間の関係図
- 時間の流れに沿ったイベントのタイムライン
- 市場構造を説明するダイアグラム
- 主張と根拠をつなぐ視覚的な構造

今はレポート生成 pipeline のテキスト品質を先に引き上げる段階だったなら、次の段階ではこのテキストをどう視覚的に補うかが重要な課題になりそうだ。

## 現時点での整理

三つのプロジェクトはそれぞれ違う方向を持っているが、全体として見れば一つの流れの中にある。

LinkVerseは自分の基盤の場所だ。  
NexLetterは現場のデータをAIで整理する自動化レポートサービスだ。  
MarketReportは市場と経済を分析し記録するコンテンツサービスだ。

それぞれのプロジェクトは独立して存在するが、結局のところ、自分が作っている世界を少しずつ違う角度から形にしている。

今の核心は、すべてを一度に完成させることではない。

各プロジェクトがどこまで来ているか、何が可能になったか、何がまだ足りないかを正確に見ることがより重要だ。

現時点から見ると、次のように整理できる。

- LinkVerseはプラットフォームの基盤と個人ブランディングの場として整理中だ。
- NexLetterはSlack Appレビュー後の開発再開を待ちながら、非定型データの処理と運用の安定性が主要な課題として残っている。
- MarketReportはレポート生成 pipeline の第1段階がほぼ完成し、いまはWebでの表現の仕方と可視化の構造が次の課題になっている。

完成したものはまだ多くはない。  
ただ、それぞれのプロジェクトがどの方向に進むべきかは、少しずつはっきりしてきている。

## 次に進めること

これからの作業は、プロジェクトごとに少しずつ違って進むだろう。

LinkVerseは当分、`/orbit`と`/blog`を中心に整理する。  
自分を紹介する場と、自分が残す記録の場を先に安定して運用してみるつもりだ。

NexLetterはSlack Appのレビューが通れば開発を続ける。  
ただそのときは単なる機能追加より、非定型のSlackデータの標準化と大量トークン処理、ワーカーの安定性、課金と billing の流れをまとめて検証する必要がある。

MarketReportは現在、ページ構成を最も積極的に考えているプロジェクトだ。  
レポート本文をどう見せるか、類似レポートをどうつなぐか、そしてテキストを補うグラフと視覚資料をどんな構造で作っていくかを整理していかなければならない。
