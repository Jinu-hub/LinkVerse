## Background

This post is a mid-point review of the Market Memory project.

Market Memory is a service I am building so that the market is not something you glance at once and move on from, but something you can accumulate judgments and interpretations about over time.

At first, the core idea was using AI to write market reports.
But as I kept working on it, I realized that simply generating reports was not enough.

What mattered was not each individual report, but the flow that emerges when those reports accumulate.

Which issues keep showing up again and again.
How the market's view of a particular technology or sector shifts over time.
How one day's market mood connects to the next day, the next week, and the next month.

That is ultimately why I named it Market Memory.
I wanted to build a structure for recording the market and being able to revisit those records later.

---

## Where Things Stand Now

As of June 8, 2026, 352 Korean-language reports have been published since December 2025.

It is still hard to call this a finished service, but the basic skeleton of the project has taken shape.

Here is what has been implemented so far.

* Report list screen
* Report detail screen
* Report search
* Basic menu structure
* Part of the Daily Market Memory dashboard
* The basic flow of the report generation pipeline
* Data accumulation based on Korean-language reports

The dashboard, Daily Market Memory, is roughly 50% complete.
Some of the core sections are in place, but many areas remain as TODOs.

The menu structure has a basic framework as well.
Still, it is hard to see the current menu layout as the final form.
As I keep building the service, some menus may be merged and others may be broken down further.

At this stage, I think it makes more sense to adjust the information structure naturally while checking how the service actually feels to use, rather than declaring a finished structure too early.

---

## What the Work So Far Means

The number 352 does not simply mean that a lot of reports have been created.

More importantly, a foundation now exists for interpreting market data in a consistent format and continuing to accumulate the results.

In particular, `core_data` is generated alongside Korean-language report writing.
This data is not just a plain text output. It is closer to an intermediate structure that can later be extended into multilingual report writing or more advanced analysis features.

This seems likely to become an important foundation for Market Memory.

Generating reports in every language from scratch from the beginning would be heavy in both cost and quality control.
On the other hand, if I can rerun only the composition pipeline based on already generated `core_data`, I can expand reports into multiple languages while keeping the same analytical structure.

In other words, the Korean reports accumulated so far are not just outputs. They are also base data for a future multilingual service and more advanced report features.

---

## Short-Term Work Still Ahead

In the short term, my goal is to reach something close to a basic release by mid-June 2026, or at the latest by mid-July.

The main tasks I am thinking about right now are as follows.

### 1. Multilingual Report Writing

The first task to tackle is multilingual report writing.

Because `core_data` is already created when Korean reports are written, there is no need to redo all the analysis from scratch.

The basic direction is as follows.

* Use the `core_data` created during Korean report generation.
* Run language-specific composition pipelines.
* Generate English and Japanese reports.
* Keep the original analytical structure, but reconstruct the content in natural sentences for each language.

This work is closer to rebuilding already organized analytical data into each language's report format than to simple translation.

If Market Memory were aimed only at Korean-speaking users, this might not be necessary.
But thinking long term, English and Japanese support would significantly broaden the service's potential.
This should be feasible with only modest changes to the automation pipeline that is already in place.

---

### 2. Multilingual UI

Alongside multilingual support for report content, the UI that displays it also needs multilingual support.

I need to organize the display across the whole interface for multilingual use: list and detail screens, menus, filters, search results, dashboard copy, and so on.

This may look like simply swapping text, but in practice there are more details to watch than it first appears.

For example, phrasing that feels natural in Korean may read a little stiff in Japanese, or become overly explanatory in English.
Date, category, tag, and summary display can also feel different by language.

At this stage, the goal is not perfect localization. For now, I want to reach a state where basic multilingual browsing and reading work without friction.

---

### 3. Daily Market Memory Dashboard Testing and Improvement

Daily Market Memory is likely to become an important entry point for Market Memory.

Reading reports one by one is still necessary, but users also need a screen where they can quickly grasp "what kind of flow the market had today" when they visit every day.

The dashboard is currently about 50% implemented.
From here, I need to check whether the sections already built actually feel meaningful.

One thing to be careful about is letting the dashboard turn into nothing more than a collection of summary cards.

Daily Market Memory should be both "today's summary" and "a record accumulating over time."
So it should show not only the current market mood, but also, little by little, what has changed compared with earlier dates.

This is something I expect to keep refining through testing.

---

### 4. Mobile Testing and Improvement

Market Memory needs basic usability on mobile as well.

A report service cannot be assumed to be read only on desktop.
There is a real chance users will read it on mobile during a commute, at lunch, or in a short break.

So I need to check basic screen display and behavior on iOS and Android.

For now, rather than aiming for app-level polish, I want to confirm that the following work properly in mobile browsers.

* Report list display
* Report detail readability
* Search
* Menu navigation
* Dashboard card display
* Scrolling and section separation in long reports
* Touch targets for buttons and links

Report content can get long, so reducing the burden of reading on mobile is especially important.

---

### 5. Vercel Deployment

I am thinking of Vercel as the deployment target.

That said, Market Memory may use more resources than expected across report generation, report viewing, and dashboard display.
At the moment, it seems likely that I will need to move to Vercel's Pro plan.

That could mean around $20 per month, but at this stage I think simplicity in development and operations matters more.

Of course, if traffic grows or server costs increase further down the line, other options may need to be considered.
But for early service validation, I think it makes more sense to deploy quickly and improve, rather than introducing overly complex infrastructure first.

---

### 6. Cron Testing and Improvement

Market Memory has data that needs to be generated on a regular schedule.

Daily Market Memory in particular needs to be generated at a fixed time each day.
So I need to confirm that cron jobs run reliably.

The areas to check are as follows.

* Whether jobs run at the scheduled time
* Whether retry or recovery is possible on failure
* Whether duplicate runs are avoided
* Whether report generation state is managed correctly
* Whether results are reflected properly on screen after generation completes

Automation is convenient, but when it fails quietly, it can be even more dangerous.
So early on, I think it is still necessary to keep some human review of automated job results.

## Medium-Term Plan Through December 2026

If the short-term goal is a basic release and stabilization, the goal through December 2026 is broader refinement of the service.

From that point on, the service needs to move beyond simply showing reports and make Market Memory's meaning clearer.

The directions I am thinking about now are as follows.

---

## Broader Service Refinement

First, the overall completeness of the service needs to improve.

Many features are still TODOs, and the menu structure may still change.
But once enough data has accumulated and the core features are stable, the next step is to give users a reason to come back repeatedly.

Simply having reports is not enough.

When users visit Market Memory, they should be able to experience the following.

* Quickly see the core flow of today's market.
* Check issues that have been appearing repeatedly in recent days.
* Explore reports related to a specific technology, company, or sector in sequence.
* See which views have strengthened or weakened over time.
* Understand the market as a flow, not as isolated news items.

I want to refine the service as a whole in this direction.

---

## Adding TODO Features

In the medium term, I want to add several features.

Representative examples include the following.

### Dashboard Refinement

Centered on Daily Market Memory, I want to build a dashboard that shows the current state of the market more clearly.

It should not be a screen that simply shows the latest reports, but one that connects past data with present data.

For example, it should be able to show today's market mood, recently strong themes, recurring risks, and differences from earlier dates more naturally.

---

### Entity Search

As the number of reports grows, simple keyword search starts to hit its limits.

It would be much more useful to explore reports by specific companies, technologies, industries, regions, people, or indicators.

For example, if users could gather related reports around entities such as "AI semiconductors," "power infrastructure," "US interest rates," "China's economy," or "NVIDIA," the browsing experience in Market Memory would improve significantly.

This feature is closer to building how reports connect to one another than to being a simple search function.

---

### Market Memory Advanced Reports

This is the direction I consider most important for Market Memory, and the feature that would best show what makes this service distinct.

Writing one good report matters, but I think the real potential of this project appears when reports accumulate.

For example, reports like the following could be created.

* How a specific issue has changed over several months
* How the market's view of a specific sector has shifted
* When a particular technology moved from a small signal to a major theme
* What risks appear repeatedly across multiple reports
* How past judgments connect to the current market situation

These reports are different from simple news summaries.
They are reports with a sense of time.

This also seems like the feature that fits the name Market Memory best.

---

## Billing Policy

I also need to settle the billing policy at some point this year.

The current thinking is simple.

Keep basic screens and some reports public, while offering advanced features and premium reports fully to paying users.

For example, I am thinking about a split like this.

* Free users: basic report access, limited search, basic dashboard
* Paid users: full report access, premium reports, advanced search, entity-based exploration, time-flow analysis reports

I have not finalized specific prices or plan structure yet.

Still, many of the features currently marked as TODOs are likely to become features offered more fully to paid users later.

I do not want to decide this too hastily.
I think it is better to decide after confirming where the service's actual value is created.

---

## Mobile Optimization and App Store Preparation

Once basic usability as a web service is stable, the next step is iOS and Android app support.

For a solo project, building separate native apps is realistically heavy.
So Market Memory is being built around the web first, with a plan to wrap that web service in a Flutter-based app wrapper for iOS / Android distribution.

A Flutter-based app wrapper is already implemented and can be tested at a basic level.
From here, I plan to check whether the web screens work reliably on mobile, while also reviewing display and usability issues when wrapped as an app.

By December 2026, I would like to have at least the following prepared.

* Mobile browser optimization
* iOS / Android screen testing
* Review of the structure needed to wrap the service as an app
* Basic information needed for app store review
* Service description copy and screenshots
* Organization around login, billing, and privacy handling

This is less a finalized plan and more a direction I am keeping open in the medium term.

---

## Current Challenges

There are also challenges I feel most strongly right now.

The first is that I still manage the source material for reports manually.

At the moment, fully automating this part is difficult.
Choosing sources for reports involves judgment beyond simple collection.

Which materials to include, which to exclude, and which perspective to prioritize even when the issue is the same.
I still think these judgments are better made by a person.

In the long term, it may become possible to automate collecting source candidates and ranking them by importance or reliability.
But at this stage, I am accepting that source management remains manual.

This is a limitation, but it is also part of the process of protecting quality.

---

## Regret Around Visual Elements

Another challenge is the visual side of reports.

Most reports today are text-centered.
I try to divide sections in Markdown and structure them for readability, but there are still moments when text alone is not enough.

Market reports are often much easier to understand when images or graphs are included.

For example, elements like the following would be helpful.

* Graphs of indicator changes
* Sector flow charts
* Frequency changes in major themes
* Mention trends for specific companies or technologies
* Risk level changes
* Visualizations of market flow by region

Especially for premium features, I would like to introduce these visual elements more actively.

That said, trying to put graphs into every report right away would make the scope too large.
So for now, I think the realistic path is to stabilize the structure of text-based reports first, then add visualization gradually in advanced features.

---

## Judgment at This Stage

Market Memory is still not a finished service.

There are many TODOs, screens still need refinement, and multilingual support remains ahead.
The dashboard is only about halfway done, and mobile testing, deployment, and cron stabilization are still needed.

At the same time, I feel the project has moved past the stage of being just an idea.

352 Korean reports have accumulated, basic report browsing and search are possible, and the shape of Daily Market Memory is starting to appear.

Most importantly, the direction this project should take is becoming a little clearer.

Market Memory is not simply a service that summarizes market news.
It is closer to a service that remembers the market over time and brings those memories back out in a form that can be analyzed again.

Right now, I am still building that structure.

## A Short Reflection

When you build a personal project, there comes a point where the list of things to do never seems to end.

At first, I spent a lot of time trying to build the report generation automation pipeline properly.
After many trials and errors, the pipeline started running to some degree,
then I needed UI to hold and present those reports.
Then I needed a dashboard, multilingual support, mobile support, deployment, and even a billing policy.

I started small, but I am reminded again that turning a service into something actually operable requires work across many layers.

Even so, this level of complexity does not feel bad right now.

If anything, it feels like a sign that the project is taking on the shape of a real service.

This is not a finished answer yet, but as of June 8, 2026, this is where my judgment stands.
For now, what matters is continuing to build on this flow.
