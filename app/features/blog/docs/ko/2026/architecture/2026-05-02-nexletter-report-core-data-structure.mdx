## NexLetter 리포트 생성 구조에 대한 새로운 고민

NexLetter를 만들면서 최근 다시 생각하게 된 부분이 있다.  
바로 **주간 리포트를 생성할 때 데이터를 어떤 단위로 처리할 것인가**에 대한 문제다.

현재 구조에서는 주간 리포트를 만들 때, 1주일 동안 수집된 데이터를 한 번에 가져와 처리하는 방식에 가깝다.

GitHub 데이터는 비교적 이 방식이 가능하다.  
커밋, PR, Issue, 변경 파일 수 등은 어느 정도 구조화되어 있고, 데이터량도 예측 가능한 편이다.

하지만 Slack 데이터는 다르다.

Slack은 현장이나 팀의 커뮤니케이션 방식에 따라 데이터량이 크게 달라진다.  
어떤 팀은 하루에도 많은 대화가 오가고, 어떤 팀은 필요한 내용만 간단히 남긴다.  
따라서 같은 “1주일치 데이터”라고 해도, 실제로 처리해야 하는 토큰량은 현장마다 크게 달라질 수 있다.

이 지점에서 `429 Rate limit reached for [모델명]`와 같은 문제가 발생할 수 있다.

물론 Rate Limit 자체를 늘리는 방법도 있다.  
하지만 그것은 OpenAI 측의 판단이나 계정 상태에 의존하는 영역이다.

서비스 설계 관점에서는 단순히 Limit 증가에 기대기보다,  
입력 데이터를 어떻게 나누고, 요약하고, 표준화하고, 단계적으로 처리할 것인지가 더 중요해질 것 같다.

---

## 기존 방식의 한계

기존 방식은 비교적 단순하다.

```txt
1주일치 Raw Data
        ↓
AI 처리 Pipeline
        ↓
주간 리포트 생성
```

MVP 단계에서는 이 방식이 나쁘지 않다.  
구조가 단순하고, 구현도 빠르다.

하지만 데이터량이 늘어나기 시작하면 몇 가지 한계가 보인다.

첫 번째는 **토큰량의 불안정성**이다.  
GitHub 데이터는 어느 정도 예측 가능하지만, Slack 데이터는 그렇지 않다.  
장애 대응, 긴 논의, 릴리즈 준비, 기획 변경 등이 겹치면 특정 주의 데이터량이 갑자기 늘어날 수 있다.

두 번째는 **처리 시간의 증가**다.  
1주일치 데이터를 한 번에 분석하고 리포트까지 생성하려면, 최종 결과를 얻기까지 시간이 길어진다.

세 번째는 **재사용성이 낮다는 점**이다.  
Raw Data를 한 번 처리하고 끝내면, 그 중간 과정에서 만들어진 분석 결과를 다른 방식으로 활용하기 어렵다.

예를 들어 나중에 일간 리포트, 월간 리포트, 기간별 유사 이슈 분석을 하고 싶어도, 다시 Raw Data부터 읽어야 할 가능성이 높다.

즉, 기존 방식은 빠르게 시작하기에는 좋지만, 데이터량이 늘어나거나 리포트의 활용 범위가 넓어질수록 한계가 드러날 수 있다.

## 개선 방향: Daily Core Data를 먼저 만든다

현재 생각하고 있는 대안은 다음과 같다.

주간 리포트를 만들더라도, 1주일치 데이터를 한 번에 처리하지 않는다.  
대신 매일 데이터를 수집하고, 그날의 데이터를 바탕으로 **Daily Core Data**를 만들어 저장한다.

그리고 주간 리포트를 생성할 때는 Raw Data 전체를 다시 처리하지 않고,  
이미 정리된 5~7개의 Daily Core Data를 기반으로 최종 리포트를 만든다.

```txt
Daily Raw Data
        ↓
Daily Core Data 생성
        ↓
Daily Core Data 저장
        ↓
5~7개의 Daily Core Data 수집
        ↓
Weekly Report 생성
```

주말에는 보통 업무가 많지 않기 때문에, 실제로는 5개 정도의 Core Data가 주간 리포트의 주요 재료가 될 가능성이 높다.

여기서 중요한 점은 Daily Core Data가 완성된 일간 리포트는 아니라는 것이다.

Daily Core Data는 최종 사용자에게 보여주기 위한 글이라기보다,  
주간 리포트나 월간 리포트를 만들기 위한 **중간 표현 데이터**에 가깝다.

다시 말하면 NexLetter의 리포트 생성 구조는 다음과 같은 방향으로 바뀌게 된다.

```txt
Raw Data
        ↓
Core Data
        ↓
Report
```

Raw Data와 Final Report 사이에 재사용 가능한 중간 데이터가 생기고,  
이 중간 데이터가 이후 확장의 핵심이 될 수 있다.

---

## Daily Core Data에 담을 수 있는 정보

현재 주간 리포트의 구성은 대략 다음과 같다.

```txt
Summary
KPI INFO
HIGHLIGHTS
TOPIC
PROGRESS AND ROADMAP
MEMBERS ACTIVITY
ENDING
```

이 중에서 `Summary`와 `ENDING`은 최종 주간 리포트 단계에서 작성하는 편이 자연스럽다.

Summary는 한 주 전체의 흐름을 보고 작성되어야 하고,  
Ending 역시 그 주의 분위기와 다음 흐름을 정리하는 역할에 가깝기 때문이다.

반면 아래 섹션들은 일간 단위로 Core Data를 만들어두기에 적합하다.

```txt
KPI INFO
HIGHLIGHTS
TOPIC
PROGRESS AND ROADMAP
MEMBERS ACTIVITY
```

예를 들어 하루 동안의 Slack / GitHub 데이터를 분석해서 다음과 같은 형태로 정리해둘 수 있다.

```txt
Daily Core Data
- 해당 날짜의 주요 KPI
- 주요 하이라이트
- 논의된 토픽
- 진행 상황과 향후 작업
- 멤버별 활동 요약
- 관련 태그
- 이슈 분류
- 데이터 출처 정보
- 사용한 모델과 프롬프트 버전
```

이렇게 만들어진 Core Data는 단순히 주간 리포트의 재료로만 쓰이지 않는다.

나중에는 특정 기간 동안 어떤 이슈가 반복되었는지,  
어떤 종류의 대응이 자주 발생했는지,  
어떤 멤버가 어떤 영역에서 활동했는지,  
특정 프로젝트의 흐름이 어떻게 바뀌었는지를 분석하는 데에도 사용할 수 있다.

---

## 기대되는 장점

이 방식의 가장 큰 장점은 처리 단위가 작아진다는 것이다.

1주일치 데이터를 한 번에 처리하는 대신, 하루 단위로 나누어 처리하면 한 번의 AI 호출에서 다루는 토큰량이 줄어든다.  
그만큼 Rate Limit에 걸릴 가능성도 낮아진다.

또한 이미 Daily Core Data가 만들어져 있다면,  
주간 리포트 생성 단계에서는 Raw Data를 다시 처음부터 읽을 필요가 없다.

정리된 핵심 데이터만 가지고 주간 흐름을 합성할 수 있기 때문에,  
리포트 생성 시간도 줄어들 가능성이 높다.

이 구조는 확장성 측면에서도 유리하다.

```txt
Daily Core Data → Daily Report
Daily Core Data 5개 → Weekly Report
Daily Core Data 20~25개 → Monthly Report
Daily Core Data N개 → 기간별 이슈 분석
```

처음에는 주간 리포트를 위해 Daily Core Data를 만들지만,  
나중에는 같은 데이터를 사용해서 일간 리포트나 월간 리포트로 확장할 수 있다.

또한 Tag와 Classification 데이터를 함께 저장해두면,  
단순한 요약을 넘어 기간별 유사 대응 체크도 가능해진다.

이렇게 보면 Daily Core Data는 단순한 캐시가 아니다.  
NexLetter가 조직의 활동 데이터를 기억하고 다시 해석하기 위한 기반 데이터가 될 수 있다.

---

## Tag / Classification이 만들어내는 또 다른 가능성

Daily Core Data 안에는 Tag나 Classification 결과도 포함시킬 예정이다.

여기서 특히 흥미로운 점은 Tag / Classification 데이터가 단순한 분류용 메타데이터에 그치지 않는다는 것이다.

예를 들어 주간 리포트 안에 특정 이슈나 토픽이 등장했을 때,  
그 이슈와 유사한 대응이 과거에 언제 발생했는지 함께 짚어줄 수 있다.

단순히 “이번 주에 이런 일이 있었다”로 끝나는 것이 아니라,  
“이와 유사한 대응은 2026년 12주차 리포트에서도 확인할 수 있다”는 식의 짧은 메모를 함께 제공할 수 있는 것이다.

```txt
참고 메모
이와 유사한 Slack 알림 실패 대응은 2026년 12주차 리포트에서도 확인할 수 있다.
당시에는 외부 API Rate Limit과 재시도 처리 방식이 주요 논점이었다.
```

이 짧은 메모 하나만으로도 리포트의 성격은 달라질것으로 생각한다.

단순히 현재 상태를 요약하는 문서가 아니라,  
과거의 대응 이력과 현재의 문제를 연결해주는 문서가 된다.

사람은 과거에 비슷한 문제가 있었다는 사실을 잊기 쉽다.  
특히 Slack이나 GitHub에 흩어진 기록은 시간이 지나면 다시 찾기 어렵다.

하지만 NexLetter가 Core Data를 기반으로 유사 이슈를 찾아주고,  
그 내용이 어떤 주차의 리포트에서 참조 가능한지까지 알려준다면,  
리포트는 단순한 요약물이 아니라 조직의 기억을 다시 꺼내주는 도구가 될 수 있다.

결국 Tag와 Classification은 검색을 편하게 하기 위한 부가 정보가 아니다.

현재의 이슈와 과거의 대응을 연결하고,  
반복되는 문제의 흐름을 파악하고,  
리포트에 더 깊은 맥락을 부여하기 위한 인덱스 역할을 하게 된다.

## 주의해야 할 점

다만 이 구조가 무조건 좋은 것만은 아니다.

가장 먼저 생각해야 할 것은 비용이다.

기존에는 주간 리포트 생성 시점에 한 번 크게 처리하면 됐다.  
하지만 Daily Core Data를 만들기 시작하면, 매일 AI 처리가 발생한다.  
그만큼 호출 횟수가 늘어나고, AI 사용료도 증가할 수 있다.

또 하나의 문제는 개발 범위다.

기존 주간 리포트 파이프라인에 Daily Core Data 생성 단계를 추가해야 하고,  
이를 저장할 테이블이나 데이터 구조도 필요하다.

또한 일간 처리에 실패했을 때의 재실행 방식, 누락 감지, 상태 관리도 함께 고려해야 한다.

중복 처리도 신경 써야 한다.

하나의 이슈가 월요일부터 수요일까지 계속 논의되었다면,  
각 Daily Core Data에는 같은 이슈가 반복해서 들어갈 수 있다.

이때 주간 리포트 생성 단계에서는 단순히 데이터를 이어붙이는 것이 아니라,  
같은 흐름을 하나의 주간 토픽으로 병합하는 과정이 필요하다.

그리고 유사 이슈를 찾아주는 기능도 신중하게 다뤄야 한다.

Tag나 Classification이 비슷하다고 해서 항상 같은 맥락의 이슈라고 볼 수는 없다.  
따라서 과거 리포트와 연결할 때는 “동일한 문제”라고 단정하기보다,  
“유사한 대응 이력이 있다”는 정도로 표현하는 것이 안전할 것 같다.

---

## 주간 리포트 단계에서 필요한 처리

Daily Core Data 기반으로 주간 리포트를 만들려면, 최종 단계에서는 단순 요약이 아니라 합성이 필요하다.

주간 리포트 생성 단계에서는 다음과 같은 처리가 중요해질 것 같다.

```txt
Weekly Synthesis
- 여러 날에 걸친 동일 토픽 병합
- 반복 이슈 제거
- 중요도 재평가
- 완료 / 진행 중 / 보류 상태 구분
- 멤버별 활동 패턴 정리
- 과거 유사 이슈 탐색
- 참고 가능한 과거 리포트 연결
- 주간 Summary 생성
- 자연스러운 Ending 작성
```

특히 Slack 데이터는 단편적인 대화가 많기 때문에,  
일간 단위에서는 중요해 보였던 내용이 주간 단위에서는 작은 이슈가 될 수도 있다.

반대로 하루 단위에서는 크게 보이지 않았던 내용이,  
며칠 동안 반복되면서 중요한 흐름으로 드러날 수도 있다.

그래서 Weekly Report 단계에서는 단순히 Daily Core Data를 합치는 것이 아니라,  
한 주 전체의 흐름을 다시 판단하는 과정이 필요하다.

여기에 과거 리포트와의 연결까지 더해진다면,  
주간 리포트는 현재의 활동을 정리하는 문서에서 한 걸음 더 나아갈 수 있다.

현재 발생한 이슈가 처음 등장한 것인지,  
과거에도 비슷한 대응이 있었는지,  
그때는 어떤 방식으로 처리했는지를 함께 보여줄 수 있기 때문이다.

## 장점과 트레이드오프 정리

정리하면 이 구조의 장점은 다음과 같다.

| 항목 | 기대 효과 |
|---|---|
| 토큰량 감소 | 한 번에 처리하는 데이터량이 줄어든다 |
| Rate Limit 리스크 감소 | 분당 토큰 처리량 부담을 낮출 수 있다 |
| 처리 속도 개선 | 주간 리포트 생성 시 이미 정리된 데이터를 사용할 수 있다 |
| 리포트 품질 향상 | Raw Data 직접 요약보다 구조화된 데이터 기반 생성이 가능하다 |
| 일간 / 월간 확장 | 같은 Core Data를 다른 기간 리포트에도 활용할 수 있다 |
| 유사 이슈 분석 | Tag / Classification을 활용해 반복되는 대응을 추적할 수 있다 |
| 과거 리포트 연결 | 현재 이슈와 유사한 과거 대응을 리포트 안에서 참조할 수 있다 |
| 재처리 용이성 | 최종 리포트만 다시 만들 때 Raw Data 전체를 다시 읽지 않아도 된다 |

반대로 트레이드오프는 다음과 같다.

| 항목 | 고려할 점 |
|---|---|
| AI 사용료 증가 | 매일 AI 처리가 발생하므로 호출 비용이 늘어날 수 있다 |
| 개발 범위 증가 | 기존 파이프라인을 꽤 크게 수정해야 한다 |
| 데이터 설계 필요 | Core Data의 스키마를 신중하게 잡아야 한다 |
| 중복 이슈 처리 | 여러 날에 걸친 동일 이슈를 병합하는 로직이 필요하다 |
| 유사 이슈 판단 | 비슷한 태그를 가진 이슈라도 실제 맥락이 다를 수 있다 |
| 운영 복잡도 증가 | 실패 재실행, 누락 감지, 상태 관리가 필요하다 |

## 현재 시점의 판단

지금 단계에서 이 구조를 바로 완성형으로 만들 필요는 없다고 생각한다.

하지만 방향성 자체는 꽤 타당해 보인다.

처음에는 단순히 Rate Limit 문제를 피하기 위한 대응처럼 보였지만,  
조금 더 생각해보면 이건 NexLetter의 구조를 더 안정적으로 만들 수 있는 방향이다.

Raw Data를 그대로 최종 리포트 생성에 넣는 방식은 빠르게 시작하기에는 좋다.  
하지만 데이터량이 늘어나고, 고객이나 현장이 다양해질수록 한계가 분명해질 가능성이 높다.

반면 Daily Core Data를 만들어두는 방식은 초기 구현 비용이 더 든다.  
AI 사용료도 증가할 수 있다.  
하지만 장기적으로는 리포트 생성 속도, 품질, 확장성, 분석 가능성 측면에서 더 좋은 기반이 될 수 있다.

특히 Tag / Classification 기반으로 현재 이슈와 과거 리포트를 연결할 수 있다면,  
NexLetter의 리포트는 단순한 주간 요약을 넘어설 수 있다.

“이번 주에 무엇이 있었는가”를 알려주는 것에서 끝나는 것이 아니라,  
“과거에도 비슷한 일이 있었는가”,  
“그때는 어떻게 대응했는가”,  
“그 내용은 어떤 리포트에서 다시 확인할 수 있는가”까지 연결할 수 있기 때문이다.

결국 중요한 것은 단순히 “리포트를 한 번 잘 만드는 것”이 아니다.

NexLetter가 계속 발전하려면,  
조직 안에서 매일 발생하는 활동 데이터를 다시 활용 가능한 형태로 축적해야 한다.

그런 의미에서 Daily Core Data는 단순한 중간 산출물이 아니라,  
NexLetter가 팀의 흐름을 기억하고 해석하기 위한 핵심 데이터가 될 수 있다.

아직 최종 결정이라고 말할 수는 없지만,  
현재 시점에서는 이 방향을 꽤 진지하게 검토해볼 만하다고 느낀다.
