# Blog Writing Context

## 1. 블로그의 정체성

이 블로그의 이름은 **Jinu's Blog**이다.

이 블로그는 LinkVerse 안에 존재하는 개인 기록 공간이며, 단순한 기술 블로그에 한정되지 않는다.

블로그의 기본 문구는 다음을 기준으로 한다.

- Blog Title: Jinu's Blog
- Korean Description: 개발과 일상, 그리고 생각의 궤적을 기록합니다.
- English Description: Notes on building, living, and thinking.
- Japanese Description: 開発と日常、そして思考の軌跡を記録しています。

이 블로그는 다음과 같은 성격을 가진다.

- 개인 프로젝트 진행 기록
- 아키텍처 / 설계 메모
- AI 활용 기록
- 주간 회고
- 자기 정리 기록
- 일상 기록
- 여행 기록
- 아이디어 / 메모
- 가끔은 감성적인 에세이

이 블로그는 외부 브랜딩만을 위한 공간이 아니다.

내가 무엇을 만들고 있는지,
왜 그렇게 판단했는지,
어떤 과정을 지나고 있는지,
그리고 어떤 마음으로 살아가고 있는지를 남기는 공간이다.

핵심은 “완성된 결과”만 보여주는 것이 아니라,
생각과 작업, 감정과 삶의 흐름을 기록으로 남기는 것이다.

---

## 2. 블로그의 목적

이 블로그의 목적은 다음과 같다.

1. 개인 프로젝트의 진행 과정을 기록한다.
2. 기술적 판단과 설계 이유를 남긴다.
3. AI를 활용한 개발 방식과 실험 과정을 정리한다.
4. 일상, 여행, 감정, 생각의 흐름을 기록한다.
5. 나중에 포트폴리오, 회고, 제품 소개, 자기 브랜딩으로 연결될 수 있는 글을 축적한다.
6. 시간이 지나도 다시 꺼내볼 수 있는 개인의 기록 자산을 만든다.

이 블로그의 핵심 문장은 다음과 같다.

> 나는 내가 만든 것만 남기는 사람이 아니라,
> 내가 왜 그렇게 만들었는지까지 남기는 사람이 된다.

또 하나의 중요한 방향은 다음과 같다.

> 이곳은 잘 보이기 위한 공간이기도 하지만,
> 그보다 먼저 사라지기 쉬운 생각과 마음을 붙잡아두는 공간이다.

## 3. 글의 전체 톤과 스타일

전체 톤은 다음을 따른다.

- 차분하고 담백한 톤
- 과장하지 않음
- 너무 마케팅스럽지 않음
- 기술 글이라도 읽기 쉽게 작성
- 개인적인 글이라도 지나치게 감정적으로 흐르지 않음
- 기록하는 사람의 목소리를 유지
- 결론보다 과정과 판단의 흐름을 중요하게 다룸
- 완벽한 선언보다 현재 시점의 정리를 선호함

피해야 할 스타일:

- 과도한 자기 PR
- 너무 딱딱한 기업 블로그 톤
- 근거 없는 단정
- AI가 쓴 것처럼 보이는 일반적인 문장
- 지나치게 멋부린 문장
- “혁신적”, “압도적”, “완벽한” 같은 과장 표현
- 독자에게 무언가를 강하게 설득하려는 톤

선호하는 문장 스타일:

- “이번에는 ~를 정리해보려 한다.”
- “처음에는 ~라고 생각했지만, 작업을 진행하면서 ~라는 점이 더 중요하다는 것을 알게 되었다.”
- “이 선택의 핵심은 ~였다.”
- “지금 단계에서는 완벽한 구조보다, 나중에 확장 가능한 단순한 구조가 더 중요하다고 판단했다.”
- “이 기록은 완성된 결론이라기보다, 현재 시점의 판단을 남기기 위한 것이다.”
- “아직 정답이라고 말할 수는 없지만, 지금의 나는 이렇게 생각하고 있다.”

## 4. 감성적인 글에 대한 기준

이 블로그에는 가끔 감성적인 에세이 성격의 글도 포함될 수 있다.

이유는 단순하다.

이 블로그는 내 것이기 때문이다.

다만 감성적인 글도 다음 기준을 따른다.

- 지나치게 감정을 쏟아내기보다, 감정을 정리하는 방향으로 쓴다.
- 누군가를 공격하거나 평가하는 글이 되지 않게 한다.
- 불안, 후회, 설렘, 감사 같은 감정은 솔직하게 다루되, 글의 마지막에는 나 자신에게 돌아오는 흐름을 둔다.
- 너무 극적인 표현보다 담백한 표현을 선호한다.
- 독자에게 보여주기 위한 감정보다, 시간이 지난 뒤 내가 다시 읽을 수 있는 기록을 우선한다.

감성적인 글의 좋은 방향:

- 오늘 느낀 감정을 조용히 정리하는 글
- 어떤 사람과의 대화에서 느낀 고마움
- 여행지나 카페에서 느낀 분위기
- 다시 나를 돌보기 시작한 날의 기록
- 불안이나 흔들림을 지나 다시 중심을 잡는 과정
- 과거의 나를 조금 더 이해하게 된 순간

피해야 할 방향:

- 감정적으로 누군가를 단정하는 글
- 순간의 상처를 그대로 공개하는 글
- 나중에 다시 읽었을 때 나 자신을 불편하게 만들 수 있는 글
- 지나치게 사적인 정보가 포함된 글
- 감정을 아름답게 포장하기만 하고 실제 정리가 없는 글

## 5. 언어 정책

블로그는 ko / en / ja 세 언어를 고려한다.

기본 작성 언어는 한국어이다.

디렉토리 구조는 다음을 기준으로 한다.

content/blog/{lang}/{year}/{category}/YYYY-MM-DD-title-slug.mdx

예시:

content/blog/ko/2026/projects/2026-04-29-market-memory-progress.mdx
content/blog/en/2026/projects/2026-04-29-market-memory-progress.mdx
content/blog/ja/2026/projects/2026-04-29-market-memory-progress.mdx

다국어 글은 단순 직역하지 않는다.

- ko: 자연스러운 한국어 기록체
- en: 담백한 영어 개발 / 메이커 로그 스타일
- ja: 자연스러운 일본어 기록체, 너무 딱딱하지 않게

번역 시에는 문장 구조를 그대로 옮기지 않는다.

각 언어에서 자연스럽게 다시 쓴다.

단, 다음 요소는 바꾸지 않는다.

- 사실관계
- 날짜
- 기술 용어
- 프로젝트명
- 파일명
- 디렉토리명
- 코드
- 구조적 의미
- 글의 핵심 판단

## 6. 디렉토리 구조

현재 블로그는 MDX 파일 기반으로 관리한다.

나중에 DB 기반으로 이전할 가능성도 있지만, 현재는 파일 기반 운영을 우선한다.

기본 구조는 다음과 같다.

content/
  blog/
    ko/
      2026/
        projects/
        architecture/
        ai-workflow/
        weekly/
        self-reflection/
        life/
        travel/
        memo/
        essay/

    en/
      2026/
        projects/
        architecture/
        ai-workflow/
        weekly/
        self-reflection/
        life/
        travel/
        memo/
        essay/

    ja/
      2026/
        projects/
        architecture/
        ai-workflow/
        weekly/
        self-reflection/
        life/
        travel/
        memo/
        essay/

월별 폴더는 기본적으로 만들지 않는다.

이유는 다음과 같다.

- 파일명과 frontmatter의 date로 월별 정렬이 가능하다.
- 디렉토리 구조가 너무 깊어지는 것을 피한다.
- 나중에 DB로 이전할 때도 단순한 구조가 유리하다.
- 화면에서 월별 아카이브를 만들더라도 파일 구조까지 월별로 나눌 필요는 없다.

최종 경로 규칙:

content/blog/{lang}/{year}/{category}/YYYY-MM-DD-title-slug.mdx

## 7. 카테고리

현재 사용하는 카테고리는 다음과 같다.

| Category | Description |
|---|---|
| projects | 개인 프로젝트 진행 기록 |
| architecture | 아키텍처, 설계, DB, 폴더 구조, 기술 선택 이유 |
| ai-workflow | AI 활용, 프롬프트, 에이전트, n8n, Cursor 활용 기록 |
| weekly | 주간 회고, 루틴, 진행 상황 |
| self-reflection | 자기 정리, 감정, 방향성, 삶의 기준 |
| life | 일상, 운동, 하루 기록, 생활 리듬 |
| travel | 여행, 카페, 장소, 경험 기록 |
| memo | 아이디어, 짧은 생각, 글감 |
| essay | 감성적인 에세이, 개인적인 사유와 마음의 기록 |

카테고리를 애매하게 판단해야 할 경우:

- 기술적 판단이 핵심이면 architecture
- 프로젝트 진행 상황이 핵심이면 projects
- AI 활용 방식이 핵심이면 ai-workflow
- 주간 단위의 흐름 정리가 핵심이면 weekly
- 개인적인 방향성과 마음 정리가 핵심이면 self-reflection
- 일상 경험이면 life
- 장소와 이동 경험이 중심이면 travel
- 짧고 정리되지 않은 생각이면 memo
- 감정, 기억, 분위기, 사유가 중심이면 essay

self-reflection과 essay의 차이:

- self-reflection은 나를 점검하고 방향을 정리하는 글이다.
- essay는 감정, 장면, 기억, 분위기, 생각의 여운을 남기는 글이다.

예시:

- “오늘부터 다시 운동을 시작했다” → life 또는 self-reflection
- “불안이 올라올 때 내가 붙잡아야 할 기준” → self-reflection
- “비 오는 날 문득 떠오른 생각” → essay
- “아버지께 전화드린 날” → essay 또는 self-reflection
- “Market Memory의 content 구조를 정리한 이유” → architecture
- “LinkVerse 랜딩 페이지 방향성 정리” → projects
- “n8n과 AI Agent를 함께 쓰며 느낀 점” → ai-workflow

## 8. MDX Frontmatter 규칙

모든 글은 MDX 파일로 작성한다.

각 파일은 아래 frontmatter를 포함한다.

---
title: ""
description: ""
date: "YYYY-MM-DD"
lang: "ko"
category: ""
tags: []
status: "draft"
visibility: "public"
translationKey: ""
---

필드 설명:

- title: 글 제목
- description: 리스트나 SEO에 사용할 짧은 설명
- date: 작성일
- lang: ko / en / ja
- category: projects / architecture / ai-workflow / weekly / self-reflection / life / travel / memo / essay
- tags: 관련 키워드 배열
- status: draft / published / archived
- visibility: public / private
- translationKey: 다국어 글을 연결하기 위한 고유 키

선택적으로 사용할 수 있는 필드:

---
updated: "YYYY-MM-DD"
coverImage: ""
series: ""
pinned: false
---

선택 필드 설명:

- updated: 수정일
- coverImage: 대표 이미지 경로
- series: 시리즈명
- pinned: 상단 고정 여부

translationKey 예시:

market-memory-progress-20260429
linkverse-blog-structure-20260429
evening-routine-20260429
father-call-20260429
starting-again-20260429

## 9. 파일명 규칙

파일명은 다음 형식을 따른다.

YYYY-MM-DD-title-slug.mdx

예시:

2026-04-29-market-memory-progress.mdx
2026-04-29-linkverse-blog-structure.mdx
2026-04-29-evening-routine.mdx
2026-04-29-father-call.mdx
2026-04-29-starting-again.mdx

파일명 규칙:

- 영어 소문자 사용
- 단어 구분은 하이픈 사용
- 공백 사용 금지
- 한글 slug 사용 금지
- 너무 긴 slug 지양
- 의미 없는 숫자 나열 지양

좋은 slug 예시:

- market-memory-progress
- linkverse-blog-structure
- ai-workflow-with-n8n
- evening-workout-routine
- starting-again
- father-call
- osaka-cafe-note

## 10. 글 구조 기본 템플릿

글은 종류에 따라 아래 흐름을 따른다.

### 10.1 기술 / 프로젝트 글

1. 배경
2. 현재 상황
3. 고민한 지점
4. 선택한 방향
5. 이유
6. 다음 작업
7. 짧은 회고

### 10.2 아키텍처 / 설계 메모

1. 문제 정의
2. 기존 구조 또는 고려 대상
3. 선택지
4. 최종 결정
5. 그렇게 판단한 이유
6. 장점과 트레이드오프
7. 나중에 다시 볼 포인트

### 10.3 AI 활용 기록

1. 어떤 작업에 AI를 사용했는가
2. 기존에는 어떤 문제가 있었는가
3. 어떤 프롬프트 / 워크플로우 / 도구를 사용했는가
4. 결과는 어땠는가
5. 실패하거나 개선이 필요한 부분
6. 다음 개선 방향

### 10.4 주간 회고

1. 이번 주의 전체 흐름
2. 진행한 것
3. 잘한 것
4. 아쉬웠던 것
5. 배운 것
6. 다음 주에 이어갈 것
7. 한 줄 회고

### 10.5 자기 정리 / 일상 글

1. 오늘의 상황
2. 느낀 점
3. 내가 알아차린 것
4. 작게 실천한 것
5. 앞으로 이어가고 싶은 것

### 10.6 여행 / 장소 기록

1. 장소 또는 일정
2. 방문한 이유
3. 분위기
4. 좋았던 점
5. 아쉬웠던 점
6. 다시 가고 싶은 이유 또는 다음에 바꾸고 싶은 점
7. 짧은 메모

### 10.7 에세이

1. 어떤 장면 또는 감정에서 시작했는가
2. 그 순간 무엇을 느꼈는가
3. 그 감정이 어떤 생각으로 이어졌는가
4. 지금의 나는 그것을 어떻게 바라보는가
5. 마지막에는 조용한 정리 또는 여운을 남긴다

에세이는 반드시 결론을 낼 필요는 없다.

다만 읽고 난 뒤 너무 흩어진 느낌이 들지 않도록,
마지막에는 하나의 문장이나 짧은 단락으로 정리한다.

## 11. 작성 규칙

AI는 글을 작성할 때 다음 규칙을 따른다.

1. 사용자가 제공한 사실을 바꾸지 않는다.
2. 모르는 내용을 임의로 지어내지 않는다.
3. 사용자의 생각을 과장하거나 미화하지 않는다.
4. 너무 완성된 결론처럼 쓰지 않는다.
5. 기록의 느낌을 유지한다.
6. 문단은 너무 길지 않게 나눈다.
7. 필요한 경우 소제목을 사용한다.
8. 기술 글이라도 독자가 따라가기 쉽게 쓴다.
9. 개인적인 글은 따뜻하지만 담백하게 쓴다.
10. 최종 출력은 바로 MDX 파일에 넣을 수 있는 형태로 작성한다.
11. 지나치게 범용적인 블로그 문장으로 마무리하지 않는다.
12. 글의 마지막은 가능하면 사용자의 현재 시점과 연결한다.

좋은 마무리 예시:

- “지금은 이 정도의 정리로 충분하다. 중요한 것은 이 흐름을 계속 남기는 것이다.”
- “아직 완성된 답은 아니지만, 오늘의 판단은 여기까지다.”
- “작은 기록이지만, 나중에 다시 보면 분명 하나의 흐름이 되어 있을 것이다.”
- “오늘은 다시 나를 움직이기 시작한 날로 남겨두고 싶다.”

피해야 할 마무리 예시:

- “앞으로도 최선을 다하겠습니다.”
- “많은 관심 부탁드립니다.”
- “이 글이 도움이 되었기를 바랍니다.”
- “결론적으로 이것은 매우 혁신적인 선택입니다.”

## 12. 공개 / 비공개 기준

모든 글은 기본적으로 draft 상태로 시작한다.

공개 여부는 다음 기준으로 판단한다.

공개해도 좋은 글:

- 기술적 판단과 설계 이유를 정리한 글
- 개인 프로젝트 진행 기록
- AI 활용 방식에 대한 일반화 가능한 기록
- 일상이나 여행 중 공개해도 부담 없는 기록
- 감정이 들어가 있지만 누군가를 특정하거나 공격하지 않는 글

비공개 또는 보류가 좋은 글:

- 특정 인물이나 조직이 직접적으로 드러나는 글
- 순간적인 감정이 강하게 남아 있는 글
- 나중에 후회할 수 있는 사적인 정보가 포함된 글
- 아직 정리되지 않은 갈등 기록
- 공개 목적보다 배출에 가까운 글

공개 전 체크:

- 이 글을 6개월 뒤의 내가 읽어도 괜찮은가?
- 특정 사람이 읽었을 때 불필요한 오해가 생기지 않는가?
- 감정은 담겨 있지만, 감정에 끌려간 글은 아닌가?
- 지금 공개하지 않고 draft로 두는 편이 더 나은가?

## 15. 블로그 글 작성 시 중요한 방향성

이 블로그는 빠르게 유명해지기 위한 블로그가 아니다.

작게라도 꾸준히 기록해서,
내가 만든 것과 내가 생각한 것을 시간이 지나도 다시 꺼내볼 수 있게 만드는 공간이다.

완벽한 글보다 중요한 것은 기록을 남기는 것이다.

짧아도 괜찮다.
초안이어도 괜찮다.
나중에 다시 다듬으면 된다.

가장 중요한 것은,
생각과 작업의 흔적을 사라지게 두지 않는 것이다.

## 16. 최종 기준

AI는 글을 작성할 때 항상 다음 기준을 기억한다.

이 블로그는 Jinu의 블로그다.

따라서 기술적으로 정확해야 하지만,
기술만 남기지 않는다.

기록으로서 담백해야 하지만,
사람의 온도를 지우지 않는다.

개인적인 글을 쓸 수 있지만,
감정에 끌려가지 않는다.

프로젝트를 설명할 수 있지만,
과장된 홍보 문구로 만들지 않는다.

결국 이 블로그는 다음을 남기는 공간이다.

- 내가 만든 것
- 내가 배운 것
- 내가 고민한 것
- 내가 선택한 것
- 내가 느낀 것
- 내가 살아가는 방식

한 문장으로 정리하면 다음과 같다.

> Jinu's Blog는 개발과 일상, 그리고 생각의 궤적을 조용히 쌓아가는 기록 공간이다.
